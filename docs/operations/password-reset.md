# パスワードリセット

スターターでは、メール送信サービス、token の保存方法、アカウント復旧のポリシーを利用者側が選べるようにするため、パスワードリセットの完全なフローを同梱しないことがあります。

## 推奨設計

- 一度しか使えない reset token を生成する。
- token はハッシュだけを保存し、user ID、有効期限、使用済み時刻を一緒に持つ。
- 生の token は、有効期限付きの reset URL としてメールで送る。
- reset 時には、token のハッシュ、有効期限、未使用かどうかを検証する。
- パスワードハッシュを更新し、認証モデルが対応していれば既存の token を無効化する。
- token そのものはログに出さず、完了イベントだけを監査ログに残す。

## 推奨エンドポイント

```text
POST /api/password/forgot
POST /api/password/reset
```

登録済みメールアドレスを推測されないようにするため、forgot-password エンドポイントは、登録の有無で区別しない共通のレスポンスを返します。

## セキュリティ上の注意点

- 有効期限は 30 分〜60 分など短めに設定する。
- メールアドレス単位と IP 単位でレート制限する。
- reset token はログに出さない。
- 使用済みの token は無効化する。
- パスワードリセット後は、既存 JWT の token version を更新して無効化することを検討する。
